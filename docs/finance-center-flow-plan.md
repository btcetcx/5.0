# 财务中心联动梳理与页面方案

日期：2026-05-31

## 目标

财务中心不要一上来铺很多页面。先做一套能承接采购、销售、售后、仓储关键财务动作的最小闭环，保证页面不乱、字段不偏、逻辑能和后端契约对上。

本方案建议财务中心第一阶段只保留 6 个核心入口：

1. 财务工作台
2. 应收管理
3. 应付管理
4. 发票管理
5. 资金核销
6. 财务设置

暂不独立做「费用报销」「成本核算」「凭证管理」「财务报表」完整模块。它们可以先在工作台和详情页中预留摘要或只读记录，等业务主链稳定后再拆出来。

## 上游业务到财务的联动

### 销售中心

来源页面：

- 客户管理
- 报价管理
- 合同管理
- 订单管理
- 销售计划

财务节点：

- 客户档案维护信用额度、账期、开票信息、财务联系人。
- 报价转合同/订单后，记录转化金额、应收状态、开票状态、回款状态。
- 合同签订后，形成合同金额、收款约定、开票记录、回款记录。
- 销售订单审核或发货后，生成应收。
- 出库/OQC/签收完成后，确认应收金额。
- 开票申请由销售发起，财务开票后回写销售订单和合同。
- 收款核销后，回写已回款、未回款、客户信用占用释放。

财务应承接：

- 应收单
- 开票申请/销项发票
- 收款单
- 核销记录
- 客户对账
- 信用占用释放记录

### 采购中心

来源页面：

- 供应商管理
- 请购管理
- 询价管理
- 采购订单

财务节点：

- 供应商档案维护财务账户、付款条件、税务信息。
- 采购订单包含订单金额、付款状态、到票状态、入库状态、三单匹配状态。
- 采购入库后，生成暂估应付。
- 到票后进行订单、入库、发票三单匹配。
- 质检扣款或退货会影响可付款金额。
- 财务付款后回写采购订单付款状态。

财务应承接：

- 应付单/暂估应付
- 付款申请
- 付款单
- 收票/进项发票
- 三单匹配记录
- 供应商对账

### 售后中心

来源页面：

- 售后单
- 售后任务
- 退货、退款、换货、维修处理

财务节点：

- 退款退货：退货入库、退款付款、应收调整、发票红冲。
- 仅退款：退款付款、发票红冲。
- 仅退货：退货入库、应收调整。
- 换货：通常不退款，但可能产生补差或应收调整。
- 维修/现场服务：可能产生服务费、配件费或免费售后成本。

财务应承接：

- 退款付款单
- 应收调整单
- 发票红冲单
- 售后费用记录
- 售后财务处理状态回写

### 仓储中心

来源页面：

- 入库管理
- 出库管理
- 库存管理
- 调拨管理
- 盘点管理
- 出库质检

财务节点：

- 采购入库完成后，影响暂估应付。
- 销售出库完成后，影响应收确认点。
- 退货入库完成后，影响应收调整、退款、红冲。
- 采购退货出库后，影响应付冲减或供应商退款。
- 盘点差异审批后，影响库存金额调整。
- 库存单价、成本层、出入库金额为成本核算来源，但第一阶段只做只读引用。

财务应承接：

- 入库来源应付
- 出库来源应收
- 退货入库来源退款/应收调整
- 盘点差异金额
- 库存成本只读记录

## 财务中心页面规划

### 1. 财务工作台

作用：只做总览和待办，不承载复杂编辑。

建议卡片：

- 待收款应收
- 逾期应收
- 待付款应付
- 待开票
- 待收票认证
- 待退款付款
- 待应收调整
- 待三单匹配
- 待银行对账

下方区块：

- 今日待办列表：来源单据、业务类型、金额、责任人、状态、操作。
- 风险提醒：逾期应收、超信用客户、到期应付、发票异常、退款超时。
- 最近财务流转：销售订单、采购订单、售后单、仓储单据的财务回写记录。

- 勾稽记录

- 核销对象
- 银行对账

关键交互：

- 收款登记
- 付款登记
- 退款付款
- 银行流水导入
- 对账匹配
- 核销到应收/应付/退款单

### 6. 财务设置

入口：

- `/finance/settings`

先只保留必要设置：

- 应收策略：应收生成节点、账龄规则、信用释放规则。
- 应付策略：暂估生成节点、三单匹配规则、付款审批规则。
- 发票策略：开票节点、红冲规则、认证规则。
- 资金策略：账户、收付款方式、核销规则、对账规则。
- 自定义字段、编号、审批、打印模板：复用公共设置母版。

不要为每个子模块都单独做一套完全不同的设置页。字段和交互必须沿用客户管理/销售设置已经验收的公共母版。

## 建议导航调整

现有老导航太多：

- 应收管理
- 应付管理
- 发票管理
- 资金管理
- 费用报销
- 成本核算
- 凭证管理
- 财务报表

建议第一阶段改成：

- 工作台：`/finance`
- 应收管理：`/finance/receivables`
- 应付管理：`/finance/payables`
- 发票管理：`/finance/invoices`
- 资金核销：`/finance/settlements`
- 财务设置：`/finance/settings`

费用、成本、凭证、报表先不作为二级导航。需要出现时：

- 成本：在仓储库存/出入库详情和财务工作台里只读展示。
- 凭证：在应收、应付、发票、资金详情 Tab 里展示「凭证记录」。
- 报表：在工作台提供简单统计，不单独做完整报表模块。
- 费用：除非用户确认要做人力/行政费用报销，否则先不做。

## 页面实现标准

1. 列表页必须使用列表母版：
   - 搜索在左。
   - 刷新、筛选、字段配置、导入、导出、新增在右。
   - 选择框、序号、操作固定列。
   - 底部有总数、已选、批量操作、分页。

2. 详情页必须使用详情母版：
   - 顶部返回、操作按钮容器。
   - 基础信息容器。
   - Tab 容器可切换，并显示对应内容。
   - 所有来源单据都能点击跳回业务来源。

3. 新增/编辑页必须使用表单母版：
   - 来源单据选择弹窗。
   - 明细子表。
   - 附件。
   - 备注/说明富文本。
   - 保存后进入列表或详情，并产生前端状态变化。

4. 设置页必须使用设置母版：
   - 自定义字段、编号、审批、策略、打印模板沿用客户管理已验收交互。
   - 不要写死不可编辑卡片。

## 避免逻辑错误的规则

1. 财务中心只承接已发生或待处理的财务动作，不反向创造业务事实。
   - 例如销售未发货，不应直接生成已确认应收。
   - 采购未入库/未到票，不应直接生成已核销应付。

2. 每条财务数据必须有来源。
   - 没有 `sourceRef` 的记录只能是手工录入，并必须标记 `sourceType = manual`。

3. 金额必须能追溯。
   - 应收金额来自销售订单/合同/出库。
   - 应付金额来自采购订单/入库/发票。
   - 退款金额来自售后处理方案。

4. 回写只回状态和汇总，不篡改业务源明细。
   - 财务可回写销售订单「已回款」「开票状态」。
   - 财务可回写采购订单「已付款」「到票状态」。
   - 财务可回写售后单「退款已付款」「应收已调整」「发票已红冲」。

5. 仓储的库存成本第一阶段只读。
   - 不要在财务中心直接修改库存数量、成本层、批次。

6. 售后关闭条件必须检查派生财务单据状态。
   - 退款退货：退货入库已完成 + 退款已付款 + 应收已调整 + 发票已红冲。
   - 仅退款：退款已付款 + 发票已红冲。
   - 仅退货：退货入库已完成 + 应收已调整。

## 推荐实施顺序

第一步：做财务工作台和导航收敛。

- 把财务中心二级导航收敛成 6 个入口。
- 工作台展示来自销售、采购、售后、仓储的待办汇总。

第二步：做应收管理。

- 优先接销售订单、销售合同、售后应收调整。
- 验证回写：销售订单已回款、客户应收未收、信用占用释放。

第三步：做应付管理。

- 优先接采购订单、采购入库、到票、付款。
- 验证回写：采购订单付款状态、到票状态、可付款金额。

第四步：做发票管理。

- 合并销项、进项、红冲。
- 验证回写：销售开票、采购收票、售后红冲。

第五步：做资金核销。

- 收款、付款、退款共用一套流水和核销模型。
- 验证回写：应收/应付/退款单核销状态。

第六步：做财务设置。

- 使用公共设置母版。
- 只放应收、应付、发票、资金四类策略。
